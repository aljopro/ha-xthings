# Xthings Home Assistant Custom Component — Copilot Instructions

## Project Overview

This is a Home Assistant custom component for **Xthings (U-tec)** smart home devices, starting with locks. The integration uses the Xthings OpenAPI (OAuth2 cloud API) to discover, control, and monitor U-tec devices.

**Goal**: Build the premier Xthings Home Assistant integration with comprehensive lock support and an extensible architecture for future device types (switches, dimmers, lights).

## Xthings API Reference

### Endpoint & Auth
- **Single endpoint**: `POST https://api.u-tec.com/action`
- **Auth**: OAuth2 Bearer token (non-standard flow: authorize URL needs client_secret, token endpoint uses GET)
- **Request format**: `{ header: { namespace, name, messageId (UUIDv4), payloadVersion: "1" }, payload: {...} }`

### Namespaces
| Namespace | Purpose |
|-----------|---------|
| `Uhome.Device` | Device discovery (`Discovery`), state queries (`Query`), commands (`Command`) |
| `Uhome.User` | User info (`Get`), unlink (`Logout`) |
| `Uhome.Configure` | Webhook registration (`Set`) — future use |

### Capabilities (from Production Capabilities doc)
| Capability | Attributes | Commands | Handler Types |
|-----------|-----------|---------|---------------|
| `st.lock` | `lockState`: locked/unlocked/jammed/unknown | `lock`, `unlock`, `setMode(mode: int)` | utec-lock, utec-lock-sensor |
| `st.BatteryLevel` | `level`: integer 1-5 | — | utec-lock, utec-lock-sensor |
| `st.healthCheck` | `status`: online/offline | — | All devices |
| `st.lockUser` | — | `list`, `get(id)`, `add(user)`, `update(user)`, `delete(id)` | utec-lock |
| `st.DoorSensor` | `sensorState`: closed/open/unknown | — | utec-lock-sensor |
| `st.Switch` | `switch`: on/off/unknown | `on`, `off` | utec-switch |
| `st.SwitchLevel` | `level`: integer 0-100 | `setLevel(level)` | utec-dimmer |

### Device Handler Types
| Handler | Capabilities |
|---------|-------------|
| `utec-lock` | Lock, BatteryLevel, LockUser |
| `utec-lock-sensor` | Lock, BatteryLevel, DoorSensor |
| `utec-dimmer` | Switch, Switch Level |
| `utec-switch` | Switch |
| `utec-light-rgbaw-br` | Switch, Brightness, Color, ColorTemperature |

### Key API Behaviors
- **Deferred responses**: Lock/unlock/setMode return `st.deferredResponse` with timeout in seconds. Poll after timeout + 2s buffer.
- **Battery scale**: Integer 1-5. Map: `{1: 10%, 2: 30%, 3: 50%, 4: 75%, 5: 100%}`
- **Case sensitivity**: Query responses use `st.Lock` (capital L) but commands use `st.lock` (lowercase). Coordinator normalizes with `.lower()`.
- **customData**: Some devices require `customData` (e.g., `userId`) to be passed back in commands. Always forward it from device discovery.

### Lock User Object Format
```json
{
  "name": "User Name",
  "type": 0,         // 0=Normal, 2=Temporary, 3=Admin
  "password": 123456, // 4-8 digit integer
  "daterange": ["2024-07-01 00:00", "2024-07-31 23:59"],  // temporary only
  "weeks": [0,1,2,3,4,5,6],  // 0=Sunday..6=Saturday
  "timerange": ["08:00", "18:00"],  // daily access window
  "limit": 1  // unlock limit; 0 or absent = unlimited
}
```
For updates, include `"id": <user_id>` in the object.

## Component Architecture

### File Structure
```
custom_components/xthings/
├── __init__.py              # Entry setup, service registration
├── api.py                   # XthingsApiClient (all API calls)
├── application_credentials.py # OAuth2 credential provider
├── config_flow.py           # OAuth2 config flow
├── const.py                 # Constants, capability strings
├── coordinator.py           # XthingsDataUpdateCoordinator (polling)
├── entity.py                # XthingsEntity base class (shared DeviceInfo, availability)
├── models.py                # XthingsDeviceInfo, XthingsDeviceState, XthingsCoordinatorData
├── lock.py                  # Lock entity platform
├── binary_sensor.py         # Connectivity, Jammed, Door sensor entities
├── sensor.py                # Battery sensor entity
├── manifest.json            # Integration metadata
├── strings.json             # UI strings, service descriptions
└── translations/
    └── en.json              # English translations (copy of strings.json values)
```

### Key Patterns

**1. Coordinator Pattern** (`coordinator.py`)
- `XthingsDataUpdateCoordinator` polls every 60s
- `_async_discover_devices()`: Caches `XthingsDeviceInfo` per device
- `_async_update_data()`: Queries each lock device, parses capabilities into `XthingsDeviceState`
- Error handling: `XthingsAuthError` → `ConfigEntryAuthFailed`, `XthingsApiError` → `UpdateFailed`

**2. Entity Registration** (each platform file)
- Filter `coordinator.data.devices` by device type (e.g., `device_info.is_lock`)
- Create entities, pass coordinator + device_info
- Entities extend `XthingsEntity` (from `entity.py`) + the HA platform base (e.g., `LockEntity`)
- `XthingsEntity` extends `CoordinatorEntity` and provides shared DeviceInfo, availability, and `_device_state`

**3. API Client** (`api.py`)
- All calls through `_async_request(namespace, name, payload)` which handles auth, UUID generation, error checking
- Lock commands: `async_lock()`, `async_unlock()`, `async_set_lock_mode()`
- User management: `async_list_lock_users()`, `async_add_lock_user()`, etc.
- Returns deferred timeout or parsed response data

**4. Deferred Command Handling**
- Lock/unlock/setMode may return timeout before executing
- Call `coordinator.async_request_refresh_after(seconds + 2)` to schedule delayed poll
- Entity shows `is_locking`/`is_unlocking` during wait

**5. Custom Services** (`__init__.py`)
- Services: `set_lock_mode`, `list_lock_users`, `add_lock_user`, `update_lock_user`, `delete_lock_user`
- Validated with `voluptuous` schemas
- Services find the right coordinator via `_get_coordinator(hass, device_id)`

### State Model
```python
@dataclass
class XthingsDeviceState:
    online: bool = False
    lock_state: str | None = None      # "locked"/"unlocked"/"jammed"/"unknown"
    battery_level: int | None = None   # Percentage (10-100)
    is_jammed: bool = False            # Derived from lock_state == "jammed"
    door_state: str | None = None      # "closed"/"open"/"unknown" (utec-lock-sensor only)
```

### Lock Entity Behavior
- Inherits from `XthingsEntity, LockEntity`
- `available`: Inherited from `XthingsEntity` — False if coordinator unavailable or no device state
- `is_locked`: True/False/None (None when lock_state is unknown)
- `is_locking`/`is_unlocking`: Set during async command execution, cleared in `finally` block
- `is_jammed`: Exposed as binary_sensor (BinarySensorDeviceClass.PROBLEM)
- `extra_state_attributes`: Includes `jammed` bool and `lock_state_detail`

## Adding New Device Types

To add support for a new device type (e.g., switch):

1. **`models.py`**: Add `is_switch` property to `XthingsDeviceInfo` checking `handle_type`
2. **`const.py`**: Add handler type constant and capability constants
3. **`coordinator.py`**: Parse new capability in `_async_update_data()` state loop
4. **`models.py`**: Add new fields to `XthingsDeviceState` (e.g., `switch_state`)
5. **Platform file** (e.g., `switch.py`): Create entity class extending `XthingsEntity` + the HA platform base (e.g., `SwitchEntity`)
6. **`__init__.py`**: Add platform to `PLATFORMS` list
7. **`api.py`**: Add command methods if device is controllable
8. **`manifest.json`**: Update version
9. **`tests/test_unit.py`**: Add test fixtures and assertions

## Testing

Tests use `importlib` to load `const.py` and `models.py` directly (avoids full HA dependency).
Run: `python3 -m pytest tests/test_unit.py -v`

For live API testing: `tests/manual_api_live.py` (requires real credentials).

### Docker Integration Testing
Use Docker to test the component in a real Home Assistant instance:
```bash
docker compose up
```
This mounts `custom_components/` into the HA container. Config is in `docker/configuration.yaml`.
After making changes, restart the container to pick them up.

## Home Assistant Custom Component Guidelines

These guidelines are derived from [integration_blueprint](https://github.com/ludeeus/integration_blueprint) and [cookiecutter-homeassistant-custom-component](https://github.com/oncleben31/cookiecutter-homeassistant-custom-component). Follow them for all HA custom component work in this repo.

### Entity Pattern
- **Always** create an `entity.py` with a base entity class inheriting from `CoordinatorEntity[YourCoordinator]`.
- Set `_attr_has_entity_name = True` in the base class.
- The base class should accept `coordinator` + device info, set up `DeviceInfo`, and provide shared properties (`available`, device state accessors).
- Platform entities inherit from **both** the base entity class and the HA platform class (e.g., `XthingsEntity, LockEntity`).
- Use `_attr_name = None` for the primary entity so the device name is the entity name. Use descriptive `_attr_name` for secondary entities (e.g., `"Battery"`, `"Connectivity"`).
- Set `_attr_unique_id` in each platform entity's `__init__` with a suffix (e.g., `{device_id}_lock`, `{device_id}_battery`).

### Coordinator Pattern
- Use `DataUpdateCoordinator` with typed data (generic parameter).
- Handle auth errors → `ConfigEntryAuthFailed`, API errors → `UpdateFailed`.
- Provide a typed `data` property returning `XthingsCoordinatorData`.
- Keep polling interval in `const.py` as `DEFAULT_SCAN_INTERVAL`.

### API Client Pattern
- Use a single API client class with typed exception hierarchy (`XthingsApiError`, `XthingsAuthError`).
- All HTTP calls go through one internal method that handles auth headers and error checking.
- Return typed data, not raw dicts.

### Config Flow
- Use `strings.json` for all user-facing text with `[%key:common::config_flow::...]` references where possible.
- Provide `translations/en.json` with resolved English strings.
- Handle reauth flow for OAuth2 integrations.

### Code Style & Linting
- **Linter**: Ruff (configured in `.ruff.toml`, targeting `py312` minimum).
- **Rule set**: `select = ["ALL"]` with targeted ignores for formatter-incompatible rules.
- Run `scripts/lint` or `ruff check . && ruff format . --check` before committing.
- Follow Home Assistant core code style conventions.

### Git Workflow
- **Branch strategy**: Work on feature branches, merge to `main` via pull requests. Never push directly to `main` for feature work.
- Branch naming: `feature/<description>`, `fix/<description>`, `chore/<description>`.
- Make small, focused commits — each commit should represent one logical change.
- Write descriptive commit messages: short summary line, then bullet points for details.
- Label PRs with the appropriate label from `.github/labels.yml` so Release Drafter categorizes them correctly.
- All CI checks (lint, validate) must pass before merging.

### File & Project Structure
- `manifest.json`: Must include `domain`, `name`, `version`, `codeowners`, `config_flow`, `documentation`, `issue_tracker`, `iot_class`.
- `hacs.json`: Must include `name`, `homeassistant` minimum version, and `render_readme`.
- `.github/workflows/validate.yml`: Combined hassfest + HACS validation.
- `.github/workflows/lint.yml`: Ruff linting + format check.
- `.github/workflows/release-drafter.yml`: Auto-drafts release notes from merged PRs.
- `.github/workflows/release.yml`: Automated release — bumps manifest version, tags, publishes GitHub release.
- `.github/workflows/labeler.yml`: Syncs repo labels from `.github/labels.yml`.
- `.github/release-drafter.yml`: Release Drafter config (categories, version resolver, template).
- `.github/labels.yml`: PR labels used by Release Drafter to categorize changes.
- `.github/ISSUE_TEMPLATE/`: `bug.yml`, `feature_request.yml`, `config.yml`.
- `scripts/`: `setup`, `develop`, `lint` bash scripts (must be executable).
- `requirements.txt`: Pin `homeassistant`, `ruff`, and other dev dependencies.
- `.ruff.toml`: Ruff linter config targeting py312+, `select = ["ALL"]` with targeted ignores.

### Releases & Release Notes
- **Release Drafter** auto-generates draft release notes on every push to `main`.
- PRs should be labeled with one of the labels defined in `.github/labels.yml` (e.g., `enhancement`, `bug`, `breaking`, `documentation`).
- Release Drafter groups changes by label into categories (Features, Fixes, Breaking Changes, etc.).
- Version is auto-resolved from PR labels: `breaking` → major, `enhancement` → minor, `bug`/`performance`/`refactoring` → patch.
- **To release**: Go to GitHub Actions → "Release" workflow → "Run workflow". Version auto-increments based on the PR labels merged since the last release. You can optionally override with a manual version.
- The workflow automatically: bumps `manifest.json` → commits → creates git tag → publishes GitHub release with the drafted notes.
- HACS picks up new versions from GitHub releases automatically.
- For manual releases: edit the draft in GitHub Releases, update `manifest.json` version yourself, then publish.

### Testing
- Use `pytest` with `pytest-homeassistant-custom-component` when possible.
- Unit tests can import `const.py` and `models.py` directly via `importlib` to avoid full HA dependency.
- Live API tests go in `tests/manual_api_live.py` (never run in CI).
- Use the Docker dev environment (`docker-compose.yml`) for integration testing against a real HA instance.

### Docker Development
- `docker-compose.yml` mounts `custom_components/` into the HA container.
- Run `docker compose up` to start a local HA instance for manual testing.
- Config is in `docker/configuration.yaml`.
